# 云上田园 · 软件架构设计文档 v2.0

> 版本：v2.0
> 更新日期：2026-04-28
> 修订人：binbin
> 适用范围：MVP 起步期 + v1 拆分演进
> 本版相对 v1 的核心变化:
>
> 1. **C 端从 Taro 双端改为 miniapp / web 双独立项目** —— 微信小程序保留 Taro,C 端 Web Portal 独立做 Vue 3 + Vite,以服务"喜欢电脑浏览的用户"和"大屏看监控更清晰"的场景
> 2. **彻底删除"直播"概念** —— 我们做的是**用户私有摄像头远程监控**(1 摄像头 → 1 用户,看自己认养地块的农作物),不是 1 推流 → N 观众的直播。砍掉 SRS / CDN 推流分发 / 观看人数统计 / 弹幕等所有直播架构
> 3. **monorepo 落地** —— 仓库切成 `apps/{miniapp,web,admin,api}` + `packages/{shared,ui-tokens,api-client}`,pnpm workspace 管理
> 4. **后端定型 NestJS + Prisma + MySQL + Redis**(可选)
> 5. **摄像头接入选型萤石云 OpenAPI**(SaaS,零运维)

### 2.3 包共享策略

| package | 内容 | 谁用 |
|---|---|---|
| `packages/shared` | TS interface(User/Package/Order/Plot/Camera...)、业务常量(套餐价格/状态枚举)、日期/手机号格式化等纯函数 | 4 个 app 全用 |
| `packages/ui-tokens` | CSS 变量(主色、阴影、圆角、字号),已在 miniapp 用 | miniapp / web 共用,admin 用 Element Plus 主题不引入 |
| `packages/api-client` | 自动生成的 axios 客户端(基于 NestJS Swagger schema) | miniapp / web / admin 全用 |

**核心收益**:类型从后端定义流到前端,改一个字段三端编译报错,不再发生"前端字段对不上后端"的 bug。

### 2.4 v1 → v2 前端重写说明

v1 假设 H5 走 Taro 同源编译。v2 拆出独立的 `apps/web/`。**`apps/miniapp/` 的代码原封保留,只是不再做 H5 编译目标**。需要重写的是 web 端,翻译规则:

| Taro 风格(miniapp 保留) | 标准 Vue 3(web 端) |
|---|---|
| `<view>` | `<div>` |
| `<text>` | `<span>` |
| `<image :src :mode>` | `<img :src class="object-fit-cover">`(mode 转 CSS) |
| `<scroll-view scroll-x>` | `<div style="overflow-x:auto">` |
| `@tap` | `@click` |
| `Taro.navigateTo` | `router.push` |
| `Taro.showToast` | 三方 toast / 自写 |

设计、布局、交互、mock 数据**完全不变**,只是把 Taro 风格"翻译"成 Web 标准。一个工程师 1-2 天可以搬完 13 个页面。

`apps/miniapp/src/main.web.js` 里的三层浏览器补丁(`<image>→<img>` MutationObserver / `@tap` 桥接 / `<scroll-view>` 增强)在 v2 后**只服务于"miniapp 临时跑 H5 调试"场景**,生产环境的 Web Portal 是 `apps/web/`,不再需要这些补丁。

### 3.2 模块 vs v1 的差异

| 模块 | v1 | v2 | 说明 |
|---|---|---|---|
| AuthModule | ✓ | ✓ | 不变 |
| UserModule | ✓ | ✓ | 不变 |
| FarmModule (套餐 + 地块混在一起) | ✓ | 拆为 PackageModule + PlotModule | 单一职责 |
| OrderModule | ✓ | ✓ | 不变 |
| ~~LiveModule~~ | ✓ | ❌ 删除 | 我们没有直播 |
| **CameraModule** | — | ★ 新增 | 私有监控 |
| CommandModule | ✓ | ✓ | 拍照指令委托给 CameraModule |
| ContentModule | ✓ | 拆为 JournalModule + PhotoModule | 田园动态 / 用户照片墙 |
| NotifyModule | ✓ | ✓ | 不变 |
| ~~FinanceModule~~ | ✓ | 推迟 | MVP 不需要,会计走人工或 SaaS |

### 3.3 单体 vs 微服务

**MVP 阶段:模块化单体**(Modular Monolith)。所有模块在一个 NestJS 进程,通过 ProviderInjection 互相调用。理由:
- 团队 < 5 人,微服务运维成本高
- 业务量第一年 < 1k 真实付费用户,单实例 2vCPU 4G 完全扛得住
- NestJS 模块边界清晰,日后想拆出去成本低

**v1 演进**:把 CameraModule 拆成独立服务(因为它要长连接萤石云 + 处理拍照任务),其他模块继续单体。

### 4.4 ORM 选型:Prisma

替代 v1 没指定的 TypeORM。理由:
- TS 类型推导更完美(从 schema 自动生成 Client 类型)
- 迁移文件用 SQL,清晰
- DX 好,新人上手快

`prisma/schema.prisma` 是数据建模的唯一来源,改字段后 `npx prisma migrate dev` 生成迁移 SQL 同时更新 Client 类型,前端 `packages/api-client` 也跟着重新生成。

## 5. 摄像头接入架构(私有监控)

> 这一节是 v2 相对 v1 改动最大的地方。**我们做的是用户私有监控,不是直播**。

### 5.1 业务逻辑澄清

| 维度 | 直播(v1 的误解) | 私有监控(v2 实际) |
|---|---|---|
| 观众 | 1 推流 → N 观众 | **1 摄像头 → 1 用户**(他自己) |
| 高并发 | 必须扛 | 不存在 |
| 推流方 | 主播 / 业务方 | **海康摄像头硬件**(4G/Wi-Fi) |
| 转码 | 必须 + CDN 分发 | 萤石云内部转一次 |
| 延迟 | 1-3 秒 | 30 秒都行(看长势,不是看比赛) |
| 用户能做 | 看 / 评论 / 抢购 | 看 / PTZ 云台控制 / "拍张照" |
| 总流量 | 巨大,需要 CDN | 极小,800 用户 × 单看几分钟/天 |

### 5.2 接入路径(选型:萤石云 OpenAPI)

```
┌────────────┐     设备扫码绑定     ┌────────────────┐
│ 海康/萤石   │─────────────────────▶│  萤石云开放平台 │
│ 摄像头      │     RTSP / 4G 流     │  (萤石 SaaS)    │
└────────────┘─────────────────────▶│                 │
                                    └────────┬─────────┘
                                             │ HTTPS / OpenAPI
                                             ▼
                                    ┌─────────────────┐
                                    │  apps/api       │
                                    │  CameraModule   │
                                    │  · 鉴权:用户能否│
                                    │    看这块地块?   │
                                    │  · 拿播放 url    │
                                    │  · PTZ 转发     │
                                    │  · 拍照抓帧 → OSS│
                                    └────────┬─────────┘
                                             │ 返回临时播放地址
                                             ▼
                            ┌────────────────────────────┐
                            │  miniapp <live-player>      │
                            │  web/admin <video> + flv.js │
                            └────────────────────────────┘
```

**对比 v1 的"自建 SRS / 腾讯云直播 LVB"**:萤石云方案**省 2 万人民币 / 年的服务器 + 5-7 天的部署调流时间**,且海康设备本身就是萤石生态原生支持。

```
 │           │                                                  │
 │       (后台 worker)                                           │
 │           │             │─出队 → 拉一帧──▶│                 │
 │           │             │                 │─返回 jpeg ─────▶│
 │           │             │                 │                  │ 上传
 │           │             │                 │◀────────────────│
 │           │             │◀────图片 url────│                 │
 │           │             │─Command(completed) + url           │
 │           │             │─发 JournalEntry 自动生成动态        │
 │ (用户在动态页看到这张照片)                                      │
```

### 5.5 关键决策

- ✅ **不自建流媒体服务器**:萤石云内置 EZOPEN 协议,小程序 `<live-player>` / 浏览器 `flv.js` 直接播
- ✅ **不存视频流**:平台不存视频,需要回看走萤石云的 7 天回看(萤石 SaaS 自带)
- ✅ **抓帧才存 OSS**:用户主动拍照才存,自动生成的生长日记图片由农技员手动上传

- 萤石云开放平台:基础免费,设备多了少量费用

- **dev**:本地 docker-compose,数据可重置
- **prod**:阿里云 ECS
- 暂不分 test / staging,人少,通过 PR 预览部署 + 手动 staging 模式

## 8. 安全与权限

### 8.1 鉴权

| 端 | 方案 |
|---|---|
| 微信小程序 | wx.login → code → 后端 code2session 拿 openid → 签 JWT (access 15min + refresh 7d) |
| C 端 Web | 短信验证码登录 → JWT(同上)。后续可加微信扫码登录 |
| Admin | 账号 + 密码 + 可选 TOTP → JWT |

JWT 黑名单存 Redis(注销时把 token 加黑名单,Guard 检查)。

### 8.2 RBAC

```
User ── role(many-to-many) ── Permission

预定义角色:
  · customer     (C 端用户)
  · agronomist   (农技员,看自己负责的地块 + 执行指令)
  · cs           (客服,查订单 + 物流 + 工单)
  · operator     (运营,管套餐 + 动态发布)
  · admin        (超管,所有权限)
```

数据级权限(ABAC):用户只能看自己的 order / adoption / camera,SQL 层面在每个查询带 `WHERE user_id = ?`,Prisma middleware 自动注入。

### 8.3 数据安全

- 传输:全链路 HTTPS,HSTS
- 存储:手机号 / 身份证字段 AES-GCM 加密,密钥存阿里云 KMS
- 密码:Argon2id
- 敏感字段脱敏展示(`138****1234`)
- OSS 文件用临时签名 url(15min 有效)

### 8.4 风控

- 限流:`@nestjs/throttler`,登录 5 次/分钟,下单 10 次/分钟,通用 100 次/分钟
- 异常登录告警(异地 / 异设备 → 短信通知)
- 摄像头查看频率限制(防爆刷萤石 API)

## 11. 架构决策记录(ADR)

| 编号 | 决策 | 理由 | 影响 |
|---|---|---|---|
| ADR-001 | MVP 用模块化单体 NestJS,不上微服务 | 团队 < 5 人,微服务运维成本高 | v1 再拆 CameraService |
| ADR-002 | 后端选 NestJS 而非 Spring Boot | 团队全 JS / TS 栈 | 若有金融级强事务可重选 |
| ADR-003 | 摄像头**不自建流媒体**,集成萤石云 OpenAPI | 成本低、稳定、海康原生 | 受限于萤石能力 |
| ADR-004 | **C 端拆分:miniapp(Taro)+ web(独立 Vue 3)** | 电脑大屏看摄像头清晰 / SEO / 不愿装小程序的用户 | 13+4 页要翻译 |
| ADR-005 | **删除直播架构** | 我们做的是私有监控,1 摄像头 1 用户,不是 1 推流 N 观众 | 不再需要 SRS / CDN / 观看人数等 |
| ADR-006 | ORM 选 Prisma 而非 TypeORM | TS 类型推导更完美,DX 优 | 团队需学习 schema 语法 |
| ADR-007 | monorepo + pnpm workspace,不上 Turborepo | 业务规模够 pnpm,Turborepo 重 | 量级上来再上 |
| ADR-008 | Redis MVP 阶段可选,不强依赖 | 简化架构,数据量小 | 真有性能问题再加 |
| ADR-009 | Admin 用 Element Plus,Web 不引第三方 UI 库 | Admin 重效率,Web 已有自写设计 token | Web 自由度高 |
| ADR-010 | 部署单机 Docker Compose | 月 200 元就够第一年 | v1 再上 K8s |
| ADR-011 | 视频流不存平台,走萤石云 7 天回看 | 节省带宽 + 合规 | 用户要老视频得跨周下单 |
